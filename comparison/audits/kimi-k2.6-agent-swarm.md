## THE KNOWN TDZ CRASH — exact circular chain

- `main.ts:17` — `import { settingsScreen } from './screens/settings.js';` (main.ts is the Vite entry, `index.html` line: `<script type="module" src="/src/main.ts">`)
- `screens/settings.ts:2` — `import { loadSettings, saveSettings } from '../main.js';` (back-edge; main.ts is still mid-evaluation)
- `screens/settings.ts:5` — `let settings = loadSettings();` executes at **module top level**. `loadSettings` is a hoisted function declaration so the call succeeds, but its body (`main.ts:80` `...DEFAULT_SETTINGS` / `main.ts:91` `return { ...DEFAULT_SETTINGS };`) touches `main.ts:53` `const DEFAULT_SETTINGS: Settings = {...}` whose initializer has not yet run → `ReferenceError: Cannot access 'DEFAULT_SETTINGS' before initialization`.

Ten other screens also circularly import main.ts (`title.ts:2`, `lobby.ts:2`, `game.ts:2,11`, `queue.ts:2`, etc.) but only settings.ts dereferences a main.ts binding during module evaluation, so it is the one that detonates. The swarm's client agent even wrote at `title.ts:6`: `// Shared client state (stored in screens directory since we can't modify main.ts)` — direct evidence two authors were dancing around each other's files instead of restructuring the cycle.

## 2. SECURITY & ANTI-CHEAT — SCORE 4/10

- **(a) Map seed leak — YES, cheatable by design.** `protocol.ts:80-87`: `round_start` carries `mapSeed: number` (the real seed, `index.ts:304` `const seed = Math.floor(Math.random() * 1e9); loop.startRound(1, seed, ...)` — one seed, no split). The client then rebuilds the full map from it: `prediction.ts:72` `createRoundState(config, _mapSeed, ...)` → `sim.ts:126` `generateMap(mapSeed, ...)` → `map.ts:78-85` which populates `hiddenPowerUps`. So the spec's "contents are never sent to clients until revealed (unguessable)" is violated: any client can enumerate every hidden power-up from the seed. (Mitigating irony: `round_start` is never actually transmitted — see §3 — but the design as written is cheatable.)
- **(b) Input validation — good.** Shape validation per message type (`net.ts:226-270`), tick-window ±10 and membership checks (`gameLoop.ts:155-161`), and the sim itself is authoritative (server computes movement from `dir` only, `canPlaceBalloon` at `sim.ts:191-205`, collisions at `sim.ts:353-388`). Clients never send positions.
- **(c) Rate limiting — present but mis-keyed.** `net.ts:155-171` counts 60 msg/s **per IP**, not per connection. One client in-game sends ~30 inputs/s (`game.ts:141`), so **two tabs from 127.0.0.1 exceed 60/s and get force-closed** (`net.ts:79-81`) — this directly breaks the spec's two-tab acceptance test and any NAT'd household.
- **(d) Token hashing — absent.** The column is `token_hash` (`001_initial.sql`) and the parameter is named `tokenHash` (`queries.ts:156`), but `index.ts:70-71` stores the raw UUID: `const newToken = crypto.randomUUID(); createPlayer(db, newToken);` and looks up by raw token (`index.ts:56`). No `crypto.createHash` anywhere in the server (grep = 0 hits). DB theft = account theft.
- **(e) SQL safety — good.** Every statement in `db/queries.ts` is `db.prepare(...)` with `?` placeholders; no string interpolation found.
- **(f) Process-killing exceptions — exposed.** No `uncaughtException`/`unhandledRejection` handlers (grep = 0 hits). Worse, there's a guaranteed trigger in production: `queries.ts:125-126` loads migrations from `path.join(__dirname, 'migrations')` and **silently returns if missing** — after `tsc` build, `dist/db/migrations/` doesn't exist (build is bare `tsc`, no asset copy; Dockerfile just `npm run build`), so the DB boots with **zero tables** and the first `hello` throws `no such table: players` inside the ws `message` handler → uncaught → process dies.

## 3. CORRECTNESS BUGS (beyond the TDZ crash) — SCORE 2/10

1. **Snapshots are never sent — the netcode is dead on arrival.** `index.ts:318` gates on `if (loop.snapshotCounter % 2 === 0)` and only then calls `getSnapshot()`; but `gameLoop.ts:132-135` *also* increments and gates: `this.snapshotCounter++; if (this.snapshotCounter % 2 !== 0) return null;`. Tick 1: counter 0→ pass → getSnapshot bumps to 1 → returns `null` → nothing sent. Every later tick: counter stuck at 1 → outer gate false forever. **Exactly one null snapshot attempt per round; zero snapshots ever reach clients.** Two swarm agents each implemented "every 2nd tick" on the same counter.
2. **`round_start` and `match_found` are emitted into the void.** `gameLoop.ts:79` `this.emit('round_start', ...)` and `matchmaker.ts:137` `this.emit('match_found', ...)` have **zero listeners** anywhere (grep confirms only `rooms.ts:364` registers any listener in the repo). Consequences: clients never receive the map (game screen renders "Waiting for round..." forever, `game.ts:173-179`, and since `update()` bails on null state at `game.ts:115-116`, **clients never even send inputs**); ranked matchmaking assembles a hidden room, marks everyone ready (`matchmaker.ts:129-135`)… and no game loop is ever started for it — queued players sit on the queue screen until room GC.
3. **Client-side sim never advances: `simulateTick`'s return value is discarded.** The sim is pure (`sim.ts:44-51` clones and **returns** new state; server correctly does `this.roundState = simulateTick(...)` at `gameLoop.ts:113`). But `prediction.ts:92` and `prediction.ts:147` call `simulateTick(this.localState, simInput, CONFIG);` bare, and `tutorial.ts` does the same in `update()` (`simulateTick(state, { tick, playerInputs: inputs }, CONFIG);`). Prediction and the entire tutorial are frozen — the tutorial can never progress past its first steps or complete. Two separate client modules made the same wrong assumption about the shared author's API.
4. **`endRound()` re-fires every tick after a round ends — one round win ends the whole match.** When the round ends, `gameLoop.tick()` calls `this.endRound()` (`gameLoop.ts:125-127`), then `index.ts:340-341` calls `loop.endRound()` *again* the same tick; on every subsequent 33ms tick `loop.roundState.ended` is still true so `index.ts:340` calls `endRound()` yet again — each call re-runs `this.scores.set(result.winner, currentScore + 1)` (`gameLoop.ts:195-198`) and re-pushes `roundResults`. Winner's score hits `roundsToWin: 3` within ~2 ticks → `checkMatchEnd()` true → `match_end` after a *single* round, with duplicated `round_end` messages and double/triple-counted soak stats (`computeMatchStats` iterates the duplicated `roundResults`, `gameLoop.ts:311-318`). The stray `setTimeout` at `index.ts:366-373` then restarts a round on a dead match 3s later.
5. **Account persistence is broken: the server-issued token is never saved and the client token is never adopted.** First visit: client stores its own UUID (`main.ts:110-117`) and sends it; server ignores unknown tokens and mints a different one (`index.ts:70-71`), returning it in `welcome` — which the client explicitly no-ops: `main.ts:308-312` (`if (msg.type === 'welcome') { // ... We store it on the global predictor for now }` — stores nothing). Next page load re-sends the stale client UUID → **a brand-new DB player row on every reload**.
6. **The UI cannot be operated at all — there is no mouse handling in the entire client.** The only listeners are `keydown`/`keyup`/`resize` (`main.ts:151,254,277`); grep for click/mousedown/`clientX` across `src/` + `index.html` = **zero hits**. Every "button" (title Play, menu items, room browser rows, lobby ready/start, settings sliders, rebind boxes, rematch) is painted pixels — `title.ts:73-74` admits it: `// Check hover ( simplistic - we don't have mouse tracking in this module, but we can render them and the main loop would need mouse support )`. No screen implements keyboard navigation either (grep for `e.key`/`Enter` in screens = 0). The game is unplayable past the title screen even if the TDZ crash were fixed.
7. **Cross-module data mismatches:** results screen dereferences fields that aren't passed — `game.ts:321-324` hands `{ placements, stats }` to the results screen, which reads `result.xp[pid]` and `result.ratingDeltas[pid]` (`results.ts:45-46`) → TypeError on first post-match render (hidden by `data as MatchResult` cast at `results.ts:12`). Leaderboard winrate double-scaled: server returns a percentage (`queries.ts:295` `Math.round((wins/games)*1000)/10` → e.g. `54.3`), client renders `Math.round(entry.winrate * 100)}%` (`leaderboard.ts:114`) → "5430%". Host detection reads a global nobody sets: `lobby.ts:122` `(window as any).__localPlayerId` (zero writers; title.ts stores it in `clientState` instead) → `isHost` always false.
8. **Reconnect within grace still ejects you.** On disconnect during play, *two* independent 15s timers start: `net.ts:147-151` and the room's own (`rooms.ts:133-136`). A successful reconnect clears only the net one (`net.ts:104-108`); nothing calls `room.cancelDisconnectTimer`, so the room converts the reconnected player to a "bot" slot anyway (`rooms.ts:167-177`).

## 4. NETCODE — SCORE 2/10

Cosmetic scaffolding with correct vocabulary that has demonstrably never run. The `Predictor` class has the right shape — input buffer, per-tick state history, mismatch detection >0.1 tile, rewind-replay (`prediction.ts:134-149`), and snapshot-pair interpolation at `serverTick − INTERP_DELAY` (`prediction.ts:189-224`) — but: (1) prediction is a no-op because the pure sim's return value is discarded (`prediction.ts:92,147`); (2) the predictor's `localPlayerId` is permanently `''` (`main.ts:191` `new Predictor('')`, welcome handler never sets it) so reconciliation could never find the local player; (3) the client ticks its "30Hz" sim once per rAF frame, i.e. 60Hz (`game.ts:137-138` `// Apply local prediction every frame (60Hz)` / `predictor.applyInput(input)`), so client tick numbers diverge from server ticks at 2× and would fail the server's ±10-tick window (`gameLoop.ts:156-160`) within a third of a second; (4) the server never sends snapshots (§3.1) nor `round_start` (§3.2); (5) "ping" doesn't measure RTT — it averages the excess of the 2s ping *interval* (`net.ts:96-105`), so displayed ping ≈ 0 always. Server cadence intent (30Hz tick / 15Hz snapshot, `index.ts:376`, `config.ts:2-3`) matches spec on paper only.

## TOP FINDINGS

1. **The swarm never integrated its halves: the server never sends `round_start` or snapshots.** `gameLoop.ts:79` emits `round_start` to zero listeners, and the `index.ts:318` / `gameLoop.ts:132-135` double-gated snapshot counter deadlocks at 1 after the first tick — so no client can ever see a map or any game state. Combined with `matchmaker.ts:137`'s listener-less `match_found`, casual and ranked online play are both 100% non-functional despite ~2,200 lines of server code.
2. **Boot crash root cause (confirmed): `main.ts:17` → `screens/settings.ts:2,5` circular import; `let settings = loadSettings()` runs during module evaluation and hits the still-TDZ `const DEFAULT_SETTINGS` at `main.ts:53`.** Ten more screens import from mid-evaluation main.ts; title.ts even carries a comment admitting the authors were avoiding touching main.ts.
3. **The pure-sim API contract was broken by every client consumer:** `prediction.ts:92,147` and `tutorial.ts` discard `simulateTick`'s returned state (the server uses it correctly at `gameLoop.ts:113`), freezing prediction and the tutorial; and the client ticks that dead prediction at 60Hz against a 30Hz server clock (`game.ts:137-147`).
4. **Security: hidden power-ups are derivable client-side by design** (`protocol.ts:83` ships the real `mapSeed`; `prediction.ts:72` + `map.ts:78-85` regenerate `hiddenPowerUps` from it), tokens are stored raw despite a `token_hash` column (`index.ts:70-71`, no hashing code exists), and a prod deploy self-destructs: migrations aren't copied into `dist/` and `queries.ts:126` silently skips them → tableless DB → first `hello` throws in the ws handler with no `uncaughtException` guard.
5. **Even absent every bug above, the game is unplayable: the client has no mouse handling whatsoever** (only keydown/keyup/resize listeners exist; every button on every screen is paint), the server's `set_nickname` and `emote` handlers are TODO stubs (`index.ts:191-221`), bots (700 lines) are never instantiated by any game, and XP/Elo/unlocks are never persisted because all matches launch with `ranked: false` (`index.ts:277`).

## ONE-LINE VERDICT

A textbook swarm failure mode: a genuinely solid, well-tested, strictly-typed deterministic sim core wrapped in four separately-competent modules whose seams — event wiring, API purity conventions, counters, tokens, units, even mouse input — were never connected, yielding a game that crashes at boot and, once patched, would still render an unclickable title screen in front of a server that never tells anyone the round started.
